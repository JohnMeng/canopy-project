Authentication
Permission Model
The Canopy REST API is used by both humans and IoT devices. There are two types of credentials, depending on whether a human user or an IoT device is making the request. Additionally, two types of authentication are supported: BASIC AUTH and Session-based AUTH.
Credential Type HTTP BASIC AUTH Session-based AUTH Username Password
User Yes Yes Chosen by user Chosen by user
Device Yes No Device UUID Device Secret Key

BASIC AUTH for Users

To use BASIC AUTH, include the Base64-encoded "username:password" in every request. For example, if your username is "leela" and your password is "P1anetExpre55", include the following header with your request: 

Authorization: Basic bGVlbGE6UDFhbmV0RXhwcmU1NQ==

Most HTTP libraries and tools can handle BASIC AUTH for you. For example:

CURL
JQUERY
PYTHON
curl -u leela:P1anetExpre55
$.ajax({
    username : "leela",
    password : "P1anetExpre55",
    ...
});
from requests.auth import HTTPBasicAuth
requests.get(url, auth=HTTPBasicAuth('leela', 'P1anetExpre55'))
BASIC AUTH for Devices
Canopy REST API requests can be made by both human users and IoT devices. When an IoT device is making the request:
  • The device's UUID is the username.
  • The device's Secret Key is the password.
For example: 
For a device with:

    - UUID: d5c827d2-1ba7-4da6-b87b-0864f0969fa8
    - Secret Key: 00ae8a3ba42fa86765c885da90a7fcaec9898acb13fc2e00

You would:

    Base64Encode("d5c827d2-1ba7-4da6-b87b-0864f0969fa8:00ae8a3ba42fa86765c885da90a7fcaec9898acb13fc2e00")

And then send this header:

    Authorization: Basic ZDVjODI3ZDItMWJhNy00ZGE2LWI4N2ItMDg2NGYwOTY5ZmE4OjAwYWU4YTNiYTQyZmE4Njc2NWM4
    ODVkYTkwYTdmY2FlYzk4OThhY2IxM2ZjMmUwMA==

Devices
Create User-Linked Devices
Request
To create one or more new devices from when authenticated as a User: 
POST /api/create_devices
{
    "quanitity" : <QUANTITY>
    "friendly_names" : [<FRIENDLY_NAME>, ...]
}
This will create new device resource(s) and assign the current User to have the "creator" role for those device(s). This will also automatically assign a UUID and Secret Key to each new device resource.
Response 
{
    "result" : "ok",
    "devices" : [ {
        "friendly_name" : <FRIENDLY_NAME>
        "device_id" : <UUID>
        "device_secret_key" : <DEVICE_SECRET_KEY>
    }, ... ]
}
Create Anonymous Device
Some deployments of the Canopy Cloud Service allow devices to be created that are not associated with any User account. These are called "anonymous devices". Each anonymous device has a user-assigned Secret Key that is used to restrict access. Anonymous devices are automatically created whenever the API call /api/my_device is made using device credentials that have never been encountered before.
List Devices
Request
To obtain a list of all devices the current User has access to: 
GET /api/me/devices?
    filter=<FILTER>
    &sort=<SORT>
    &limit=<LIMIT>
Details about filtering, sorting, and paginating the results are described in the following sections.
Filtering Device List

<FILTER> is a sequence of constraints encoded as a string. The terms have the form: 

<KEY><REL><VALUE>
For example: 
temperature>=50.8
Unary and boolean operators can be applied to these terms using reverse polish notation: 
<TERM> <OP>
<TERM> <TERM> <OP>
For example: 
temperature>=50.8 temperature<90.5 AND
Valid terms may be the name of any cloud variable, or one of the following builtins:
Key Description Potential Values
ws_connection_status Is websocket connected?
  • "never_connected"
  • "connected"
  • "disconnected"
activity_status Has device communicated with remote recently
  • "newly_created"
  • "active"
  • "inactive"
<CLOUD_VAR_NAME> Filter based on cloud variable value. Depends on cloud variable datatype.
Valid relations are:
Relation Description
= Equals
!= Does not equal
> Greater than (numeric and date types only)
>= Greater than or equal to (numeric and date types only)
< Less than (numeric and date types only)
<= Less than or equal to (numeric and date types only)
Valid unary operators are:
Unary Op Description
NOT Invert operand
HAS For cloud variables only: does the device have a cloud variable with this name?
Valid boolean operations are:
Boolean Op Description
AND Both operands are true
OR Either operand is true
Sorting Device List
<SORT> is a sequence of comma-separated keys specifying the sort order. The highest priority key comes first, followed by tie breakers. By default ascending order is used. To sort by a key in descending order, add :DESC to the key. For sort order determination, a missing cloud variable comes before the minimum value (i.e. at the front of the list when ascending, at the end of the list when descending). For example: 
temperature,humidity:DESC
The valid keys may be any cloud variable, or builtin. See Filtering Device List.
Limits (Pagination)
By default,
Response 
{

    "devices" : [ {
        "device_id" : <UUID>,

Get Single Device
Request
To obtain information about a single device: 
GET /api/device/<UUID>
For convenience, you can obtain information about the currently authenticated device (if any) using: 
GET /api/my_device
This second form of the request can only be made when device credentials are provided. If the provided device credentials are unknown to the remote, then a new anonymous device is created.
Response 
{
    "result" : "ok",
    "device_id" : <UUID>,
    "friendly_name" : <FRIENDLY_NAME>,
    "status" : {
        "ws_connected" : <WS_CONNECTED>,
        "last_activity_time" : <LAST_ACTIVITY_TIME>,
    },
    "sddl" : { <SDDL> },
    "vars" : { <VARS> },
}

Update Device Properties
Request
To update device properties: 
POST /api/device/<UUID>
{
    "friendly_name" : <NEW_FRIENDLY_NAME>,
    "location_note" : <NEW_LOCATION_NOTE>
}

All fields are optional and omitted fields will be left unchanged.

This may be combined with Add/Update Cloud Var Metadata and Update Cloud Var Values into a single request.

Add/Update Cloud Var Metadata
Request

To add a new cloud variable, or update an existing cloud variable's metadata, use: 

POST /api/device/<UUID>
{
    "sddl" : {
        <VAR_DECL> : <VAR_PROPERTIES>,
        ...
    }
}

This may be combined with Add/Update Cloud Var Metadata and Update Cloud Var Values into a single request.

To fully replace the list of cloud variables, add "full-replacement" : true to the request. 

POST /api/device/<UUID>
{
    "sddl" : {
        "full-replacement" : true,
        <VAR_DECL> : <VAR_PROPERTIES>,
        ...
    }
}

Update Cloud Var Value

To update Cloud Variable values, use: 

POST /api/device/<UUID>
{
    "vars" : {
        <VAR_NAME> : <NEW_VALUE>,
        ...
    }
}

This will automatically create a cloud variable for each newly-encountered var, unless the device's "auto-var-creation" property is set to false. See Automatic Cloud Variable Creation.

This request may be made by either a User or a Device. The Cloud Variable's direction property controls who can modify it:

Direction Who Can Modify Value
out
in
inout
Synchronization
The "synchronization" mechanism allows multiple resources to be tracked and updated efficiently with a single API request. Tracked resources can include accounts, devices, and cloud variables. If using Session-based AUTH, the list of tracked resources will be remembered across requests.
Track Account Resources
Request 
POST api/sync
{
    ...
    "track-accounts" : [<USERNAME>, ...]
    ...
}
Response
The response will include basic information about each device that the tracked accounts have access to. 
{
    ...
    "accounts" : {
        <USERNAME> : {
            "devices" : [ <DEVICE_UUID>, ... ]
        }
    },
    "devices" : {
        <DEVICE_UUID> :  {
            "friendly_name" : <FRIENDLY_NAME>,
            "connected" : <CONNECTED>,
            "sddl_hash" : <SDDL_HASH>,
        }
    },
    "sddl" : {
        <SDDL_HASH> : {
            <SDDL>
        }
    }
    ...
}
Track Device Resources 
POST api/sync
{
    ...
    "track-devices" : [<DEVICE_UUID>, ...]
    ...
}
Response
Detailed information about tracked devices, including Cloud Variable values, will be included in the response. 
{
    ...
    "devices" : {
        <DEVICE_UUID> :  {
            "friendly_name" : <FRIENDLY_NAME>,
            "connected" : <CONNECTED>,
            "sddl_hash" : <SDDL_HASH>,
            "vars" : <VARS>,
            "notifs" : <NOTIFICATIONS>,
        }
    },
    "sddl" : {
        <SDDL_HASH> : {
            <SDDL>
        }
    }
    ...
}
